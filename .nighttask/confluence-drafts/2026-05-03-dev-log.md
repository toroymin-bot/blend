## BUG-011 — React #418 REGRESSION 수정

### 원인 분석

전 nighttask QA에서 `/ko/qatest`, `/en/qatest`, `/design1/ko/qatest` 진입 직후 + 메뉴 네비게이션마다 #418 발생.

추적 결과 **3 곳에 같은 패턴**의 lazy initializer가 있었음 — useState 초기 함수가 SSR(`typeof window === 'undefined'`)과 client(`window.innerWidth` / `localStorage`)에서 서로 다른 값을 반환.

| 파일 | 변수 | SSR 반환 | Client 첫 렌더 반환 |
|------|------|---------|--------------------|
| `src/hooks/use-device-class.ts:21` | deviceClass | `'desktop'` | `getDeviceClass(innerWidth)` |
| `src/modules/settings/settings-view-design1.tsx:157` | activeSection | `'api'` | `innerWidth < 768 ? null : 'api'` |
| `src/modules/documents/documents-view-design1.tsx:179` | tab | `'library'` | localStorage 값 (`chat` 가능) |

특히 `useDeviceClass`는 2026-04-30 v3.2 commit에서 의도적으로 lazy 도입 — 다운스트림 race 차단을 위해 hydration mismatch 위험을 알면서도 트레이드오프했었음. 이번에 그 트레이드오프가 #418로 표면화.

### 수정

**`use-device-class.ts`** — SSR-safe 복귀:
```ts
const [deviceClass, setDeviceClass] = useState<DeviceClass>('desktop');
useEffect(() => {
  function update() { setDeviceClass(getDeviceClass(window.innerWidth)); }
  update();
  // resize / orientationchange 리스너
}, []);
```

**`settings-view-design1.tsx`** — `firstMountRef` 패턴:
```ts
const [activeSection, setActiveSection] = useState<SectionId | null>('api');
const firstMountRef = useRef(true);
useEffect(() => {
  if (firstMountRef.current) {
    firstMountRef.current = false;
    if (deviceClass === 'mobile' && activeSection === 'api') setActiveSection(null);
    return;
  }
  // 이후 리사이즈 — 사용자 의도된 'api' 보존
}, [deviceClass]);
```

**`documents-view-design1.tsx`** — post-mount localStorage 반영:
```ts
const [tab, setTab] = useState<'library' | 'chat'>('library');
useEffect(() => {
  const stored = localStorage.getItem('d1:docs-tab');
  if (stored === 'chat') setTab('chat');
}, []);
```

## BUG-012 — html `lang="en"` hardcoded 수정

### 의사결정

Next.js 권장 패턴은 `[lang]/layout.tsx`에 `<html lang={params.lang}>` 두는 것이지만, Blend 프로젝트는 `output: 'export'` + `[lang]` 외부 라우트 5+개(`/qatest`, `/pricing`, `/privacy`, `/terms`, `/refund`, `/oauth-callback`)가 있어 root layout 분리 시 충돌 발생. 큰 리팩터 부담.

대안: **root layout `<body>` 최상단에 inline blocking script** 추가 — pathname 매칭 시 `document.documentElement.lang` 즉시 보정. 첫 페인트 전 동기 실행되므로 스크린리더 + DOM lang API 모두 올바른 값 본다. SEO도 모던 검색 엔진(Google 등)이 JS 실행하므로 무난.

### 수정

```tsx
// src/app/layout.tsx body 최상단
<script
  dangerouslySetInnerHTML={{
    __html: `(function(){try{var m=location.pathname.match(/^\\/(?:design[123]\\/)?(ko|en)(?:\\/|$)/);if(m)document.documentElement.lang=m[1];}catch(e){}})();`,
  }}
/>
```

### 검증

- 로컬 dev `/design1/ko` → `document.documentElement.lang === 'ko'` ✅
- 로컬 dev `/design1/en` → `document.documentElement.lang === 'en'` ✅
- production HTML grep 결과 inline 스크립트 정상 포함

## PUBLISH-PY-IMPROVE

전 nighttask 시작 시 publish.py가 hardcoded PAGES 만 처리해 5/2 파일 인자를 무시하고 5/1 페이지를 5/2 내용으로 덮어쓴 사고가 있었음. 재발 방지.

### 변경

```python
# 사용법
python3 publish.py                       # auto-pick newest YYYY-MM-DD-dev-log.md
python3 publish.py 2026-05-03            # 해당 날짜 dev-log
python3 publish.py 2026-05-03-dev-log.md # 파일명 직접
python3 publish.py file1.md file2.md     # 다중 파일
```

- `sys.argv` 인자 받으면 그것만 publish
- 인자 없으면 `YYYY-MM-DD-dev-log.md` 패턴 + 파일명 날짜 + mtime 정렬 → newest 1개만
- 첫 H1 (`# Title`) 라인에서 제목 자동 추출, fallback 은 `Blend 개발일지 — YYYY-MM-DD (새벽 nighttask)`
- Hardcoded PAGES 제거
